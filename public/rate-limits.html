Rate Limits

Rate limits protect the API from abuse and ensure fair usage across all clients. Limits are applied per API key and reset on a rolling window basis.

Rate limit headers

Every API response includes headers that tell you your current rate limit status:

HeaderDescription
X-RateLimit-Limit Maximum requests allowed in the current window
X-RateLimit-Remaining Requests remaining before you hit the limit
X-RateLimit-Reset Unix timestamp when the window resets
Retry-After Seconds to wait before retrying (only on 429 responses)
Response Headers
X-RateLimit-Limit: 5000
X-RateLimit-Remaining: 4987
X-RateLimit-Reset: 1741824060
Content-Type: application/json

Handling 429 errors

⚠
When you exceed your rate limit, the API returns 429 Too Many Requests. Always implement retry logic with exponential backoff.
JSON — 429 Response
{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "Too many requests. Please retry after 60 seconds.",
    "retry_after": 60
  }
}
Node.js — Retry with backoff
async function fetchWithRetry(url, options, retries = 3) {
  const res = await fetch(url, options);

  if (res.status === 429 && retries > 0) {
    const retryAfter = res.headers.get('Retry-After') || 60;
    await new Promise(r => setTimeout(r, retryAfter * 1000));
    return fetchWithRetry(url, options, retries - 1);
  }

  return res;
}

Tips to stay within limits

ℹ
Cache responses — avoid fetching the same data repeatedly. Store results locally and refresh only when needed.

Batch requests — use list endpoints with filters instead of fetching individual resources in a loop.

Use webhooks — instead of polling for changes, subscribe to webhook events to get notified in real-time.
